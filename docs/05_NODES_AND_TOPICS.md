# Архитектура ROS 2: Узлы, Топики и Интерфейсы (Mona Robot)

В данном документе описана графовая архитектура проекта Mona. Система построена по модульному принципу (Component-based) с жёстким разделением логики управления, навигации и безопасности.

## 1. Домены и Категории

### 1.1. Безопасность и FDIR (Ядро системы)
Отвечает за перехват управления, маршрутизацию потоков и аппаратную безопасность.
- **Узлы:**
    - `/safety_node` (C++ Component) — Интеллектуальный мультиплексор и контроллер безопасности. Управляет переключением между автопилотом и человеком (таймаут 5 секунд), а также EMA-сглаживанием.
    - `/mona_fdir_manager` (Python) — Менеджер отказоустойчивости. Следит за "пульсом" узлов и принимает решения о перезагрузке компонентов (Soft/Hard Reboot).
- **Ключевые топики:**
    - `/system/health_state` (`std_msgs/String`) — Текущий уровень здоровья системы (NORMAL, DEGRADED, EMERGENCY и т.д.).
    - `/robot_status` (`std_msgs/String`) — Режим работы мультиплексора (AUTONOMOUS, MANUAL, IDLE, NO_CMD_DATA).
    - `/hardware/contactor_cmd` (`std_msgs/Bool`) — Команда на физическое замыкание/размыкание реле питания моторов.
- **Сервисы:**
    - `/emergency_stop` (`std_srvs/Trigger`) — Жёсткая остановка (Level 2 E-STOP).

### 1.2. Ручное управление (Teleop)
- **Узлы:**
    - `/joy_node` — Драйвер геймпада (DualSense / Xbox).
    - `/teleop_twist_joy_node` — Транслятор кнопок геймпада в векторы скорости.
- **Топики:**
    - `/joy` (`sensor_msgs/Joy`) — Сырые данные со стиков и кнопок (100 Гц).
    - `/cmd_teleop` (`geometry_msgs/Twist`) — Команды управления от человека (Имеют наивысший приоритет в `safety_node`).

### 1.3. Навигация и Планирование (Nav2)
- **Узлы:**
    - `/controller_server` — Локальный планировщик (DWB). Отслеживает траекторию, учитывает динамические препятствия.
    - `/planner_server` — Глобальный планировщик. Строит маршрут по статической карте.
    - `/bt_navigator` — Координатор поведений на базе Behavior Trees.
    - `/smoother_server`, `/velocity_smoother`, `/waypoint_follower`.
- **Топики:**
    - `/goal_pose` (`geometry_msgs/PoseStamped`) — Целевая точка назначения.
    - `/cmd_nav` (`geometry_msgs/Twist`) — Команды скорости от автопилота (Блокируются на 5 сек при касании геймпада).
    - `/plan` / `/local_plan` (`nav_msgs/Path`) — Визуализация проложенного маршрута.

### 1.4. Восприятие (Perception)
- **Узлы:**
    - `/mona_lidar_merger` (C++ Component) — Сливает облака точек с 4-х лидаров в единое пространство (Zero-Copy).
    - `/pointcloud_to_laserscan` — Конвертирует 3D-облако обратно в 2D-скан для Nav2.
- **Топики:**
    - `/lidar_{front/back/left/right}/scan` — Сырые данные лидаров из симуляции/железа.
    - `/perception/combined_cloud` (`sensor_msgs/PointCloud2`) — Склеенное 3D-облако.
    - `/scan` (`sensor_msgs/LaserScan`) — Финальный 2D-скан (360 градусов), который читают Costmaps.

### 1.5. Локализация и Кинематика (EKF & SLAM)
- **Узлы:**
    - `/ekf_filter_node` — Сливает одометрию колёс и данные IMU.
    - `/slam_toolbox` — Строит карту и позиционирует робота на ней.
    - `/robot_state_publisher` — Публикует дерево TF (преобразования координат) на основе URDF.
- **Топики:**
    - `/odom` — Сырая одометрия (от колёс).
    - `/odom_filtered` — Сглаженная одометрия (с учётом IMU).
    - `/map` (`nav_msgs/OccupancyGrid`) — Карта склада.
    - `/tf` и `/tf_static` — Дерево трансформаций робота.

## 2. Практическое руководство: Как этим пользоваться

Для отладки логики робота нет необходимости запускать интерфейсы. Вы можете имитировать поведение систем напрямую из терминала.

### 2.1. Управление FDIR и Безопасностью

`fdir_manager` постоянно пишет статус системы в топик `/system/health_state`. Вы можете перехватить этот топик и вручную перевести робота в любой режим.

**Тестирование деградации (Медленный режим):**
Если датчик сбоит, FDIR шлёт `DEGRADED`. Робот должен резко снизить максимальную скорость. Проверить это:
```bash
ros2 topic pub --once /system/health_state std_msgs/msg/String "{data: 'DEGRADED'}"
```

**Тестирование E-STOP (Обесточивание моторов):**
Имитация нажатия красной кнопки на корпусе или критического сбоя.
```bash
ros2 topic pub --once /system/health_state std_msgs/msg/String "{data: 'EMERGENCY'}"
```
_Ожидаемый результат:_ Робот встанет как вкопанный, в терминал прилетит `HARDWARE CONTACTOR CUTOFF!`, Nav2 будет заблокирован.

**Восстановление работы:**
```bash
ros2 topic pub --once /system/health_state std_msgs/msg/String "{data: 'NORMAL'}"
```

### 2.2. Управление Жизненным циклом (Lifecycle)

Главные узлы (`safety_node`, `mona_lidar_merger`, и весь стек Nav2) — это управляемые Lifecycle-узлы.

**Узнать текущее состояние узла:**
```bash
ros2 lifecycle get /safety_node
```
_(Ожидаемый ответ: `active`, `inactive` или `unconfigured`)_

**Вручную отключить обработку данных (Pause):**
Если вам нужно, чтобы робот перестал реагировать на управление, но не выключался полностью, переведите узел безопасности в режим "Неактивен" (Сработает `on_deactivate`):
```
ros2 lifecycle set /safety_node deactivate
```
_Ожидаемый результат:_ Узел остановит публикацию в `/cmd_vel`, железо (контакторы) будет программно отключено.

**Вернуть в работу:**
```bash
ros2 lifecycle set /safety_node activate
```
Каждый раз при смене состояния нода публикует отчёт. Посмотреть, кто и когда менял статус:
```bash
ros2 topic echo /safety_node/transition_event
```

### 2.3. Имитация движения (Без джойстика)

Вы можете двигать робота напрямую, эмулируя работу геймпада или автопилота.

**Сигнал от телеоператора (Включает автоматическую 5-секундную блокировку Nav2):**
```bash
ros2 topic pub --rate 50 /cmd_teleop geometry_msgs/msg/Twist "{linear: {x: 0.5, y: 0.0}, angular: {z: 0.0}}"
```

**Сигнал от Автопилота:**
```bash
ros2 topic pub --rate 10 /cmd_nav geometry_msgs/msg/Twist "{linear: {x: 0.5, y: 0.0}, angular: {z: 0.0}}"
```

_Примечание:_ Если вы сначала подадите сигнал в `/cmd_teleop`, а затем сразу в `/cmd_nav`, второй сигнал будет проигнорирован благодаря логике `manual_timeout_` (5 секунд после ручного управления блокирует другие источники управления `cmd`).

### 2.4. Проверка железа

Хотите убедиться, что логика безопасности реально обесточивает контакторы моторов? Слушайте этот топик:
```bash
ros2 topic echo /hardware/contactor_cmd
```
- `data: true`  -> Реле замкнуто, на мостовые драйверы моторов подано питание (24V/48V).
- `data: false` -> Реле разомкнуто, моторы обесточены (сработает аппаратный тормоз).
