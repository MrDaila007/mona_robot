# Архитектура подсистемы SLAM и Одометрии

## 1. Обзор пайплайна
Подсистема картографирования построена на базе стека ROS 2 Humble с использованием `slam_toolbox` (режим online asynchronous). Для обеспечения математической стабильности алгоритмов SLAM применяется строгая иерархия трансформаций (TF) и высокочастотная фильтрация сенсорных данных.

### 1.1. Дерево трансформаций (TF Tree)
Система поддерживает классический стандарт REP-105:
`map` -> `odom` -> `base_footprint` -> `base_link` -> `[sensor_links]`
* **map -> odom**: Публикуется узлом `slam_toolbox`. Компенсирует дрейф одометрии на основе сопоставления лазерных сканов (Scan Matching).
* **odom -> base_footprint**: Публикуется узлом `ekf_node` (Extended Kalman Filter пакета `robot_localization`). Работает на частоте 50 Гц.
* **base_footprint -> base_link**: Статическая трансформация (подошва робота на Z=0 к физическому центру масс Z=0.17).

## 2. Источники данных и обработка

### 2.1. Идеальная одометрия (Ground Truth)
Из-за ограничений плагина `MecanumDrive` в Ignition Gazebo (пропуски таймстемпов, коллизии типов сообщений и отсутствие публикации ковариации), одометрия рассчитывается через плагин `OdometryPublisher`. Это обеспечивает идеальную частоту (50 Гц) и точные абсолютные координаты робота в симуляции, предотвращая математический коллапс в SLAM. 
*Важно: Топик `/tf` строго исключен из моста `ros_gz_bridge`, чтобы избежать конфликтов (Race Condition) с узлом EKF.*

### 2.2. Обработка данных Лидара
Тяжелый робот (200 кг) оснащён 4 лидарами (развертка каждого 150°), встроенными в ниши корпуса.
1. **Слияние**: Данные с 4 лидаров сливаются в единое облако `PointCloud2` узлом `mona_lidar_merger`.
2. **Проекция (pointcloud_to_laserscan)**: 3D-облако проецируется в 2D `LaserScan`.
   * **Target Frame**: `base_footprint` (исключает ошибки задержки статического TF при старте).
   * **Use Inf**: `True` (критично для инициализации свободного пространства).

## 3. Настройка SLAM Toolbox
Используется решатель **Ceres Solver** (SPARSE_NORMAL_CHOLESKY).
* **scan_buffer_maximum_scan_distance**: 40.0m (адаптировано под масштаб склада).
* **minimum_travel_distance / heading**: 0.1m / 0.1rad (обновление карты только при физическом смещении робота).
* **Очередь сообщений**: `scan_buffer_size: 30` (защита от кратковременных задержек TF).

## 4. Сохранение и сериализация карты
Сохранение артефактов для стека Nav2 производится через официальный графический плагин `SlamToolboxPlugin` в RViz2.
Артефакты сохраняются в директорию `src/mona_core/maps/`.
* **Save Map** (`.pgm`, `.yaml`): Создает 2D-сетку занятости (Occupancy Grid). Используется сервером карт Nav2 для планирования маршрутов.
  `/home/vladubase/mona_ws/src/mona_core/maps/warehouse_map`
* **Serialize Map** (`.posegraph`, `.data`): Сохраняет внутренний математический граф поз SLAM. Позволяет возобновить картографирование (Lifelong Mapping) после перезапуска системы без потери предыдущих данных.
  `/home/vladubase/mona_ws/src/mona_core/maps/warehouse_map_serialized`
