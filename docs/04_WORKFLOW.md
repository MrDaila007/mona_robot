# Ежедневный рабочий процесс (Daily Workflow)
В этом документе описаны типовые команды, которые разработчик выполняет каждый день внутри контейнера.
## 1. Сборка (Build)
В проекте используется `colcon` - стандартный инструмент сборки ROS 2.
### Полная сборка
Собирает все пакеты в рабочей области. Используйте это, если меняли код в нескольких пакетах сразу.
```bash
colcon build --symlink-install
````
### Сборка конкретного пакета (Быстро)
Если вы работаете только над одним пакетом (например, `mona_safety`), нет смысла пересобирать всё.
```bash
colcon build --packages-select mona_safety --symlink-install
```
### Проблемы со сборкой (Clean Build)
Если возникают ошибки CMake или проблемы с линковкой:
1. **Вариант А: Сброс кэша (Рекомендуется)** Используйте этот флаг, если вы изменили `CMakeLists.txt` или добавили новые файлы, но старый кэш мешает сборке.
```bash
colcon build --cmake-clean-cache --symlink-install
```
2. **Вариант Б: Полная очистка (Nuclear Option)** Если ничего не помогает, удалите папки сборки полностью:
```bash
rm -rf build/ install/ log/
colcon build --symlink-install
```

---
## 2. Обновление окружения (Source)
После **каждой** успешной сборки необходимо обновлять окружение, чтобы ROS увидел новые файлы.
```bash
source install/setup.bash
```
> **Совет:** Если вы добавили новую ноду или Launch-файл, и ROS её не видит — скорее всего, вы забыли сделать `source`.

---
## 3. Тестирование (Local Testing)
Перед отправкой кода (Push) обязательно запустите тесты локально. Это сэкономит время на CI.
```bash
# Запуск тестов для конкретного пакета
colcon test --packages-select mona_safety

# Просмотр результатов (если есть ошибки)
colcon test-result --verbose
```

## 4. Управление зависимостями (rosdep)

### Зачем это нужно в C++?
В отличие от Python, где зависимости изолируются в виртуальном окружении, C++ зависимости в ROS 2 - это **системные библиотеки** (заголовочные файлы `.hpp` и бинарные `.so` в `/usr/lib`).
Когда вы добавляете `<depend>lifecycle_msgs</depend>` в `package.xml`, CMake не скачивает библиотеку сам. Он ожидает, что она уже установлена в системе. Утилита `rosdep`:
- Читает `package.xml`.
- Видит там `<depend>lifecycle_msgs</depend>`.
- Идет в свою базу данных и смотрит: "В Ubuntu этот пакет называется `ros-humble-lifecycle-msgs`".
- Вызывает `sudo apt-get install ros-humble-lifecycle-msgs`.
### Когда запускать?
Эту команду нужно выполнять, если:
1. Вы клонировали проект в первый раз.
2. Вы (или ваш коллега) добавили новую библиотеку в `package.xml` и проект перестал собираться с ошибкой вида `Could not find a package configuration file provided by...`.
### Команда
```bash
# --from-paths src: сканировать папку src/
# --ignore-src: не пытаться установить пакеты, которые мы сами же и разрабатываем
# -r: продолжать установку даже при ошибках
# -y: отвечать "yes" на все вопросы apt-get

rosdep update && rosdep install --from-paths src --ignore-src -r -y
```
