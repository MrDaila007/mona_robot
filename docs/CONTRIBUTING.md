# Правила участия (Contributing Guidelines)

> **Quality Gate & Governance**
> Каждый коммит — это потенциальное изменение поведения физического устройства весом в сотни килограмм. Качество кода — вопрос безопасности.
## 1. Git Flow
Мы используем ветвление на основе задач. Прямые коммиты в `main` запрещены.
### Именование веток
* `feat/name` — новая функциональность.
* `fix/name` — исправление ошибки.
* `refactor/name` — изменение кода без изменения логики.
* `docs/name` — обновление документации.
### Сообщения коммитов
Следуйте спецификации **Conventional Commits**:
* `feat: add lidar safety check`
* `fix: resolve null pointer exception in navigation`
* `docs: update installation instructions`
## 2. Процесс Pull Request (PR)
1. Убедитесь, что `./scripts/local_ci.bash` проходит успешно.
2. PR должен содержать описание изменений.
3. PR должен быть проверен минимум одним другим разработчиком (Code Review).
4. CI (GitHub Actions) должен быть зелёным.
## 3. Стандарты безопасности и архитектуры (Safety & Compliance)
Архитектура MONA проектируется с учётом требований индустриальных стандартов для автономных мобильных роботов (AMR).
### 1. ISO 3691-4 (Поведение и среда)
*Главный стандарт безопасности для "Driverless Industrial Trucks".*
* **Влияние на код:** Определяет логику переключения зон безопасности лидаров, дистанции торможения и лимиты скорости в зависимости от окружения.
* **Реализация:** Навигационный стек должен поддерживать динамические карты ограничений (Costmaps) и зоны замедления.
### 2. ISO 13849-1 (Надёжность управления)
*Стандарт функциональной безопасности (Performance Levels - PL).*
* **Влияние на код:** Требует реализации механизмов диагностики сбоев и отказоустойчивости.
* **Реализация:**
    * **Watchdog Timers:** Все управляющие узлы обязаны иметь таймеры сброса (Dead man's switch).
    * **Lifecycle Management:** Использование управляемых нод (Managed Nodes) для гарантии корректной инициализации драйверов.
    * **QoS:** Использование политики `Reliability: Reliable` для критических топиков (E-Stop, /cmd_vel).
### 3. VDA 5050 (Интероперабельность)
*Стандартный интерфейс связи AGV/AMR с облачным сервером (Master Control).*
* **Влияние на код:** Унификация машины состояний робота и протоколов обмена данными (MQTT).
* **Реализация:** Архитектура `mona_core` должна позволять легкую интеграцию `mqtt_bridge` для трансляции ROS 2 сообщений в формат VDA.

Следуем общему алгоритму:

**Шаг 1. Получение актуальной версии проекта**
```bash
git checkout main
git pull origin main
````

**Шаг 2. Создание новой ветки**
Назовите ветку понятно, используя префиксы выше.
```bash
git checkout -b feat/my-new-feature
```

**Шаг 3. Сохранение изменений**
```bash
# Добавляем все изменённые файлы или конкретные файлы
git add .
# ИЛИ
git add src/my_cool_pkg

# Создаём коммит
git commit -m "feat: implement basic velocity controller"
```

**Шаг 4. Отправка на сервер**
```bash
git push -u origin feat/my-new-feature
```

После этого перейдите на GitHub и нажмите кнопку **"Compare & pull request"**. Или более старший разработчик увидит ваш Pull Request и проверит ваши изменения.
После этого онлайн запустятся тесты CI и при их успехе ваши изменения смержаться в ветку `main`.
