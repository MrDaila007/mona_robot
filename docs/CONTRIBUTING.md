# Правила разработки (Contribution Guidelines)
Этот документ описывает процессы взаимодействия в команде и стандарты качества, принятые в проекте MONA.
## 1. Стратегия веток (Git Flow)
Мы не коммитим напрямую в `main`. Все изменения проходят через Pull Request (PR).
### Именование веток
Используем префиксы:
* `feat/` - Новая функциональность (фича) (пример: `feat/lidar-driver`)
* `fix/` - Исправление ошибок (пример: `fix/cmake-path`)
* `docs/` - Обновление документации
* `chore/` - Настройка CI, Docker, зависимостей
* `refactor/` - Улучшение кода без изменения логики
### Стандарты коммитов
Используем Conventional Commits: `<тип>: <описание>`
* `feat`: add safety lifecycle node
* `fix`: resolve cmake install path error
* `docs`: update readme installation guide

## 2. Процесс сдачи кода
Следуем общему алгоритму:

**Шаг 1. Получение актуальной версии проекта**
```bash
git checkout main
git pull origin main
````

**Шаг 2. Создание новой ветки**
Назовите ветку понятно, используя префиксы выше.
```bash
git checkout -b feat/my-new-feature
```

**Шаг 3. Сохранение изменений**
```bash
# Добавляем все изменённые файлы или конкретные файлы
git add .
# ИЛИ
git add README.md docs/01_SETUP.md

# Создаём коммит
git commit -m "feat: implement basic velocity controller"
```

**Шаг 4. Отправка на сервер**
```bash
git push -u origin feat/my-new-feature
```

После этого перейдите на GitHub и нажмите кнопку **"Compare & pull request"**. Или более старший разработчик увидит ваш Pull Request и проверит ваши изменения.
После этого онлайн запустятся тесты CI и при их успехе ваши изменения смержаться в ветку `main`.

## 3. Стандарты качества и Индустрия
Проект MONA разрабатывается с прицелом на промышленные стандарты складской робототехники.
### Языки и Стиль
- **C++ 17**: Основной язык для алгоритмов, драйверов и Safety-логики.
    - Следуем [ROS 2 C++ Style Guide](https://docs.ros.org/en/humble/The-ROS2-Project/Contributing/Code-Style-Language-Versions.html).
    - Используем `rclcpp::Node` и `LifecycleNode`.
- **Python 3.10**: Используется **только** для Launch-файлов, тестов и вспомогательных скриптов.
### Промышленные протоколы (Target Architecture)
#### VDA 5050 (Communication Interface)
Мы стремимся к совместимости со стандартом **VDA 5050**.
- **Что это:** Стандартный интерфейс связи между роботами (AGV/AMR) и системой управления флотом (Master Control) через MQTT.
- **Зачем:** Чтобы MONA могла работать в одном складе с роботами других производителей (KUKA, Omron и т.д.).
- **Реализация:** В будущем будет добавлен `mona_vda5050_bridge`, транслирующий ROS 2 топики в JSON MQTT сообщения.
### Безопасность (Safety Standards)
При разработке узлов безопасности (Safety Node) мы ориентируемся на требования:
- **ISO 3691-4**: "Industrial Trucks - Driverless industrial trucks and their systems". Это главный стандарт безопасности для складских мобильных роботов.
- **ISO 13849-1**: Описывает уровни производительности (PL) для систем управления безопасностью.
> **Важно:** Любой код, отвечающий за движение робота, должен иметь механизмы Fail-Safe (остановка при потере связи, ошибки сенсора или падении процесса).
### Валидность данных
В проекте будет использоваться большое число датчиков, у каждого из которых есть свои слабые стороны:
- Физика датчика (слепая зона, маленькая дистанция, чувствительность к блестящим и зеркальным поверхностям).
- Потеря связи.
- Шумы.
**Главная задача** - исключить шумные или заведомо неправильные входные данные и помечать их, или исключать полностью из расчётов.
Здесь нам в помощь стандартная библиотека C++ `std:optional`, `std:variant`, `[[nodiscard]]`, `concept`.
